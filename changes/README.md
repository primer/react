# changes

A Rust CLI for managing the release process of packages in the `primer/react` monorepo.

## Overview

`changes` wraps the [changesets](https://github.com/changesets/changesets) file format (`.changeset/*.md`) and provides:

| Command | Purpose |
|---|---|
| `changes add` | Create a new changeset describing which packages changed and how |
| `changes version` | Apply pending changesets: bump package versions and update changelogs |
| `changes publish` | Publish packages to npm, create git tags, and create GitHub releases |
| `changes release` | Create a release branch, version packages, and open a pull request |
| `changes status` | Show pending changesets and the version bumps they will produce |

## Installation

```sh
# From the repo root
cargo install --path changes
```

Or build and run in place:

```sh
cargo build --release --manifest-path changes/Cargo.toml
./changes/target/release/changes --help
```

## Usage

### `changes add`

Create a new changeset interactively (prompts for package selection, bump type, and description):

```sh
changes add
```

Or supply all flags for use in scripts / CI:

```sh
changes add \
  --package @primer/react \
  --bump minor \
  --message "Add new Dialog size variants"
```

Multiple packages can be listed by repeating `--package`:

```sh
changes add \
  --package @primer/react \
  --package @primer/styled-react \
  --bump patch \
  --message "Fix TypeScript types"
```

---

### `changes version`

Apply all pending changesets, bump the matching package versions, update each package's `CHANGELOG.md`, and remove the consumed changeset files:

```sh
# Version all affected packages
changes version

# Version only a specific package
changes version --package @primer/react
```

---

### `changes publish`

Publish packages that have an unpublished local version to npm, create a git tag for each (`<name>@<version>`), and create an autogenerated GitHub release.

```sh
# Publish everything that needs publishing
changes publish

# Publish a single package only
changes publish --package @primer/react

# Use a different npm dist-tag
changes publish --tag next

# Preview what would happen
changes publish --dry-run
```

`$NODE_AUTH_TOKEN` must be set to a valid npm token, and `$GITHUB_TOKEN` must be set for GitHub release creation (via the `gh` CLI).

---

### `changes release`

Creates a new branch `release/YYYY-MM-DD`, runs `changes version`, commits the result, pushes the branch, and opens a pull request to `main`.

```sh
# Standard release (all packages with pending changesets)
changes release

# Release only specific packages
changes release --package @primer/react

# Custom date (default: today)
changes release --date 2025-01-28

# Cherry-pick commits onto the release branch before versioning
changes release --cherry-pick abc1234 --cherry-pick def5678

# Skip opening the PR (useful for CI that opens it with a custom token)
changes release --no-pr

# Dry run – see what would happen without making any changes
changes release --dry-run
```

Requires the `gh` CLI to be installed for PR creation.

---

### `changes status`

Show pending changesets and the version bumps they will produce:

```sh
changes status
```

## Release Process

The release process is automated through two GitHub Actions workflows:

### Weekly release (`.github/workflows/changes-prepare-release.yml`)

Runs every Monday at 10:00 UTC. It:
1. Creates a `release/YYYY-MM-DD` branch.
2. Runs `changes version` to apply pending changesets.
3. Commits the version bumps and removes the changeset files.
4. Pushes the branch and opens a pull request to `main` labelled `release`.

Can also be triggered manually from the **Actions** tab with options to:
- Release a specific subset of packages.
- Provide a custom date.
- Cherry-pick specific commits (for hotfixes/patch releases).
- Run in dry-run mode.

### Publish on merge (`.github/workflows/changes-publish.yml`)

Triggers automatically when a `release/*` branch PR is merged into `main`. It:
1. Builds all packages (`npm run build`).
2. Runs `changes publish` to push packages to npm.
3. Creates a git tag per package (`<name>@<version>`) and pushes them.
4. Creates an autogenerated GitHub release for each tag.

Can also be triggered manually with options to publish specific packages, use a custom npm dist-tag, or dry-run.

## Changeset file format

Changesets are stored in `.changeset/<id>.md` and follow the standard format:

```md
---
"@primer/react": minor
---

Brief description of the change that will appear in the changelog.
```

Supported bump types: `patch`, `minor`, `major`.

Both double-quoted and single-quoted package names are accepted when parsing.

---

## Cherry-picking for patch releases

To backport a fix onto a release branch after it has been created:

```sh
# On the main branch, commit your fix
git commit -m "fix: correct button border radius"

# Create a new release targeting a single package,
# cherry-picking the fix commit
changes release \
  --package @primer/react \
  --cherry-pick <commit-sha> \
  --date 2025-01-28-patch1
```

Or, via the manual `workflow_dispatch` trigger on `changes-prepare-release.yml`, fill in the **cherry_picks** input with a comma-separated list of SHAs.
